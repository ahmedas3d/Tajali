## Summary

Redesign the Settings screen into a fully branded, grouped preference hub (welcome card + three sections: الصلاة والمواقيت / القرآن الكريم / عام) and upgrade the adhan notification engine to support per-prayer on/off toggles, two-day-ahead scheduling on every app open, and a new Fiqh school (Madhab) preference that immediately recalculates Asr. All option pickers use modal bottom sheets. The preferred Quran reciter setting in Settings drives the `selectedReciterProvider` already wired to the Quran reader. No new packages are needed — every dependency is already in `pubspec.yaml`.

## Key Design Decisions

### Per-Prayer Toggle Storage
Each prayer toggle is a separate `bool` SharedPreferences key. On startup all 5 are loaded and passed via `overrideWith` to 5 individual `StateProvider<bool>` instances. This makes each toggle independently reactive with no serialization overhead.

### 2-Day-Ahead Scheduling
`adhanSchedulerProvider` already watches `prayerTimesProvider`. It is extended to also calculate tomorrow's prayer times (using `PrayerCalculationService.rawTimes` with `date + 1 day`) and collect entries for both days. `buildEntries` already skips past times, so today's already-passed prayers are filtered out automatically. Only prayers with their toggle set to `true` are included.

### Madhab Wiring
`PrayerCalculationService.calculate()` and `rawTimes()` gain an optional `Madhab madhab` parameter (default `Madhab.shafi`). The `adhan` package's `CalculationParameters.madhab` field is set before constructing `PrayerTimes`. `prayerTimesProvider` watches `fiqhSchoolProvider` and maps `FiqhSchool → Madhab`.

### Settings Screen Architecture
`SettingsScreen` is a full `ConsumerWidget` rewrite. Option pickers call `showModalBottomSheet` inline — no separate routes. The `SettingsPermissionBanner` widget checks `Permission.notification.status` via `AppLifecycleListener` (Flutter 3.13+) and auto-hides when the user returns after granting permission.

### Reciter Bridge
`selectedReciterProvider` lives in `lib/features/quran/providers/reader_providers.dart`. `SettingsScreen` imports it directly and calls `saveReciter()` from the same file. No duplication.

## Implementation Phases

### Phase A — Engine upgrades (no UI, independently testable)
1. Add `FiqhSchool` enum + `fiqhSchoolProvider` to `prayer_times_providers.dart`
2. Add 5 `prayerNotifProvider`s + `loadSaved*` / `save*` helpers; wire into `main.dart`
3. Extend `PrayerCalculationService` with optional `madhab` parameter
4. Extend `AdhanNotificationService`: per-prayer cancel-by-id + 2-day scheduling loop
5. Update `adhanSchedulerProvider` to pass madhab + per-prayer filter

### Phase B — Settings screen redesign
6. Build widget library: `settings_welcome_card`, `settings_section_header`, `settings_value_row`, `settings_toggle_row`, `settings_link_row`, `settings_permission_banner`
7. Rewrite `settings_screen.dart` using new widgets + inline `showModalBottomSheet` calls
8. Build `about_screen.dart` (static content: version, developer info)

### Phase C — Integration & wiring
9. Wire reciter picker in Settings to `selectedReciterProvider` + `saveReciter()`
10. Wire permission banner: `AppLifecycleListener` → re-check `Permission.notification.status`
11. Verify all new keys loaded in `main.dart` with correct `overrideWith` calls
12. Smoke test: run app, cycle through all settings, verify notifications fire
