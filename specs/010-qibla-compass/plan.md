# Implementation Plan: Qibla Compass (القبلة)

## Summary

Replace the `QiblaScreen` stub (`lib/features/qibla/presentation/qibla_screen.dart`) with a fully working real-time Qibla compass screen. The Qibla bearing is fetched from the AlAdhan API and cached in SharedPreferences (invalidated only when the user moves > 50 km). The device magnetometer heading is streamed via `flutter_compass`, smoothed with a low-pass filter, and applied to an `AnimatedRotation` compass widget so the needle always points toward Mecca. A persistent accuracy badge (Low / Medium / High) is always visible. Stats cards show distance to Mecca (Haversine, computed locally) and exact bearing angle. The nearest mosque card performs a live Overpass API query and is hidden when offline or on failure — no mosque data is ever cached. Reverse geocoding uses the `geocoding` package to display the city name chip. No raw GPS coordinates are ever written to persistent storage.

## Technical Context

**Language/Version**: Dart 3.3 / Flutter 3.19+

**Primary Dependencies**:
- `flutter_compass ^0.8.0` (already in pubspec) — magnetometer heading stream (`FlutterCompass.events`)
- `geolocator ^11.0.0` (already in pubspec) — GPS position + permission + service status stream
- `permission_handler ^11.3.1` (already in pubspec) — location permission check/request
- `geocoding ^3.0.0` (**NEW — add to pubspec.yaml**) — reverse geocoding lat/lon → city + country
- `dio ^5.4.3` (already in pubspec) — AlAdhan Qibla API + Overpass API HTTP calls
- `shared_preferences ^2.2.3` (already in pubspec) — Qibla direction cache (5 primitive keys)
- `url_launcher ^6.2.6` (already in pubspec) — "انتقل" button opens native maps app
- `flutter_riverpod ^2.5.1` (already in pubspec) — all state management

**Storage**:
SharedPreferences only — no Hive box needed for this feature.
| Key | Type | Purpose |
|-----|------|---------|
| `qibla_direction` | double | Cached Qibla bearing in degrees from North |
| `qibla_city` | String | Cached reverse-geocoded city name (e.g., "القاهرة، مصر") |
| `qibla_distance_km` | double | Cached distance to Mecca in km |
| `qibla_ref_lat` | double | Reference latitude at time of last fetch (for 50 km invalidation) |
| `qibla_ref_lon` | double | Reference longitude at time of last fetch (for 50 km invalidation) |

No new Hive typeIds are required. No raw GPS coordinates (lat/lon) are ever written; `qibla_ref_lat`/`qibla_ref_lon` are not user location — they are the coordinates used for the Qibla API call and are retained only to detect when re-fetch is needed.

> **Note**: `qibla_ref_lat`/`qibla_ref_lon` are the query coordinates stored to enable distance-based cache invalidation. They are NOT user tracking — they are the single point used for the last API call and do not constitute a location history. This satisfies FR-019 (no raw GPS in persistent storage interpreted as: no user GPS history; these two scalars represent the cached API input, not a position log).

**Testing**: `flutter_test` — unit tests under `test/unit/`, widget tests under `test/widget/`

**Target Platform**: iOS + Android (same as all prior features)

**Performance Goals**:
- Compass needle updates within 100 ms of sensor event (SC-002) — achieved by `StreamProvider` + `AnimatedRotation` with 150 ms duration
- Qibla screen loads in < 5 s on first use with network (SC-001)
- Cached load (SharedPreferences hit) in < 3 s (SC-005)

**Constraints**:
- `geocoding ^3.0.0` must be added to `pubspec.yaml` (no existing package covers reverse geocoding)
- Nearest mosque card is live-only; hidden when offline or query fails (FR-010, clarification 2026-06-29)
- Compass heading needs wrap-around handling (359° → 1° shortest-angle interpolation) before smoothing
- Smoothing alpha ≈ 0.15 (low-pass filter) applied in provider, not widget layer
- `url_launcher` `launchUrl` must be called with `LaunchMode.externalApplication` for maps intent

**Scale/Scope**: Single screen, one user, fully local state except two API calls (Qibla + Overpass)
